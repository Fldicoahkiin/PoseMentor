# 几何关系与相机参数说明

## 1. 这份文档解决什么问题

本项目里，素材同步、多机位 2D 预览、3D 融合、三角化真值生成，依赖两类参数：

- 时间参数：不同机位在同一动作时间轴上的偏移量
- 几何参数：相机内参、畸变、外参，以及统一世界坐标系约定

两类参数缺一不可：

- 只有时间对齐，没有几何参数，只能做同步播放和多机位 2D 观察
- 只有几何参数，没有时间对齐，无法把同一时刻的 2D 点稳定融合为 3D

## 2. 当前 AIST++ 使用的参数来源

AIST++ 预览对齐层使用以下官方文件：

- `data/raw/aistpp/annotations/aist_plusplus_final/cameras/mapping.txt`
  - 作用：把每个动作序列映射到对应的相机方案 `setting*.json`
- `data/raw/aistpp/annotations/aist_plusplus_final/cameras/setting*.json`
  - 作用：提供每个机位的内参、畸变、旋转、平移
- `data/raw/aistpp/annotations/aist_plusplus_final/keypoints2d/*.pkl`
  - 作用：提供 9 机位官方 2D 关键点轨迹
- `data/raw/aistpp/annotations/aist_plusplus_final/keypoints3d/*.pkl`
  - 作用：提供官方 3D 时间轴参考

当前实现会基于官方 3D 参考和相机参数，把 3D 关键点重投影到各机位图像平面，再与官方 2D 做误差搜索，得到每个机位相对于公共时间轴的 `offset_frames`。

## 3. 当前系统里的参数含义

### 3.1 时间对齐参数

- `timeline_fps`
  - 公共时间轴帧率
- `timeline_frame_count`
  - 官方时间轴的总帧数
- `timeline_start_frame`
  - 当前预览实际采用的公共时间轴起点
- `camera_offsets[camera_id]`
  - 机位相对于公共时间轴的偏移
  - 含义：同一时刻 `t`，该机位应读取 `t + offset` 的源帧
- `camera_trim_start[camera_id]`
  - 实际导出预览时，从该机位原视频裁切的起始帧
- `frame_total`
  - 当前这组可同步播放的公共有效帧数

### 3.2 几何参数

每台相机至少包含：

- `image_size`
  - 图像分辨率，例：`[1920, 1080]`
- `intrinsic`
  - 内参矩阵 `K`
- `distortion`
  - 畸变系数，通常是 5 维或更多
- `rotation`
  - 旋转参数，当前 AIST++ 为 Rodrigues 向量
- `translation`
  - 平移向量

项目在前端展示的 `focal_length_px` 与 `principal_point_px` 是从 `intrinsic` 里直接展开出的摘要信息：

- `fx = K[0,0]`
- `fy = K[1,1]`
- `cx = K[0,2]`
- `cy = K[1,2]`

## 4. 自定义四机位最少必须提供什么

后续如果接入自定义四机位数据，最少需要以下内容：

### 4.1 必须项

1. 一组动作对应多路视频
   - 同一动作必须能明确配成同一组
2. 稳定的机位命名
   - 例如 `front / left / right / back` 或 `c01 / c02 / c03 / c04`
3. 每路视频的实际帧率与分辨率
4. 时间同步依据
   - 最低要求：统一拍手、统一起跳、统一口令中的任意一种
5. 相机内参与外参
   - `K`
   - `distCoeffs`
   - `R` 或 Rodrigues `rvec`
   - `t`
6. 统一坐标系定义
   - 哪个方向是前后
   - 哪个方向是左右
   - 哪个方向是竖直向上

### 4.2 强烈建议项

1. 固定场地、固定三脚架、固定焦段
2. 分辨率统一，建议全程 1080p
3. 帧率统一，建议 60 FPS
4. 每次开拍前保留 1~2 秒静止段
5. 每次机位移动后，重新记为新的标定版本

### 4.3 不建议让拍摄人员承担的事

以下步骤更适合后处理脚本完成：

- 统一转码
- 补齐命名
- 时序偏移估计
- 2D 关键点提取
- 预览裁切和缓存
- 三角化质量统计
- 重投影误差筛选

## 5. 自定义数据的推荐交付结构

```text
session_001/
  front.mp4
  left.mp4
  right.mp4
  back.mp4
  calibration.json
  session.json
```

### 5.1 `session.json` 最小字段

```json
{
  "session_id": "session_001",
  "action_id": "arabesque_01",
  "performer_id": "p001",
  "take_id": "take_01",
  "fps": 60,
  "resolution": [1920, 1080],
  "camera_names": ["front", "left", "right", "back"]
}
```

### 5.2 `calibration.json` 最小字段

```json
{
  "name": "studio_a_rig_01",
  "world_unit": "meter",
  "cameras": {
    "front": {
      "image_size": [1920, 1080],
      "intrinsic": [[1200, 0, 960], [0, 1200, 540], [0, 0, 1]],
      "distortion": [0.01, -0.02, 0.0, 0.0, 0.0],
      "rotation": [[1, 0, 0], [0, 1, 0], [0, 0, 1]],
      "translation": [0.0, 1.6, 4.0]
    }
  }
}
```

说明：

- 当前项目多机位三角化模块读取的是统一 `calibration` 结构
- 自定义数据如果遵循这一结构，后续可直接复用现有三角化与预览逻辑

## 6. 哪些是系统可以自动算的，哪些不能省

### 6.1 可以自动化

- 根据拍手或动作突变估计粗同步
- 统一导出分辨率与帧率
- 自动提取 2D 关键点
- 自动生成同步预览
- 自动统计重投影误差、骨长稳定性、缺帧情况

### 6.2 不能省

- 同一动作的多机位配对关系
- 基础命名规范
- 至少一次可靠标定
- 同步依据
- 固定的世界坐标定义

## 7. 前端里展示这些参数的目的

前端新增的“对齐参数”区域主要用于回答三个问题：

1. 当前这组素材到底按哪套相机方案在播
2. 每个机位被裁掉了多少帧，为什么不是都从第 0 帧开始
3. 这组素材当前能稳定同步多少帧，误差大概是多少

这块不是只给开发看，后续也可以直接给采集团队或标定人员确认：

- 是否用了正确的机位方案
- 哪个机位偏移异常大
- 哪个机位的同步误差明显偏高

## 8. 当前实现边界

当前已经落地：

- AIST++ 官方多机位时间对齐元数据生成
- 基于官方相机参数的多机位预览裁切
- 前端展示对齐参数、机位偏移、焦距与主点摘要

当前还未覆盖：

- 自定义四机位的自动标定求解
- 自定义数据的硬件级同步
- 多人身份关联
- 面向大规模生产的标定版本管理

这些后续都可以沿用同一套参数结构继续扩展。
